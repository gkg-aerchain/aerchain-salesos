The Aerchain Pricing Model

A unified, defensible pricing framework for autonomous procurement — purpose-built for enterprise deals ranging from $100M to $10B+ in spend under management.

Executive Summary

Aerchain's pricing model is a unified three-layer framework: a spend-tier anchored subscription (platform access + workflow fees), a soft multiplier layer for user and transaction volume, and a smooth volume-discount power curve. Every component is deterministic code — not AI, not opinion. A $250M mid-market deal and a $3B global enterprise deal are priced from the exact same rules, with every assumption traceable to a single configuration file. The model is designed to be defensible to a CFO, explainable to a procurement VP, and tunable by a sales leader — without code changes.

1Pricing Philosophy

Three principles govern how Aerchain prices:

1. Spend is the anchor

The fundamental pricing driver is annual spend under management, not user count or module SKUs. A customer with $1B of addressable spend receives a different price than one with $500M, regardless of headcount. This matches the value Aerchain delivers: the platform's economics scale with the dollars flowing through it, not the seats accessing it.

2. Deterministic math, not AI

All pricing calculations are pure, reproducible code. Given identical inputs, the calculator produces identical outputs — every time. AI is used for parsing RFP context and generating narrative, never for the numeric math itself. This guarantees auditability and zero hallucination risk on client-facing numbers.

3. Configurable, not hardcoded

Every parameter — spend tiers, multipliers, discount curves, channel weights, implementation rates — lives in a structured JSON config, not in code. Sales leadership can tune the model directly via the in-app Pricing Rules panel without developer involvement. Changes are immediate and visible.

4. Transparent, layered composition

Every dollar of the final Y1 total can be traced back to a specific component: Platform Access, Workflow Fees, Integration Add-Ons, Implementation Services, and (optionally) Gain Share. Clients see exactly what they're paying for and why — no bundled opacity.

4Platform / Workflow Split (45/55)

Once the net baseline is computed, it's split into two component layers by a configurable ratio (default 45% Platform Access + 55% Workflow Fees):

Platform Access (45%)

The recurring fee for access to the Aerchain platform: AI models, infrastructure, standard integrations (ERP, SSO), security and compliance, customer success, and SLA commitments. This is the fixed base — the client pays this regardless of how much they actually use the platform.

Scales with: user count relative to tier benchmark (soft ±15% multiplier)

Workflow Fees (55%)

The variable fee for workflow execution: sourcing events, autonomous negotiations, contract authoring, PO orchestration, invoice matching. This represents the per-transaction value of running the platform at scale.

Scales with: actual monthly transaction volume relative to expected (soft [0.85x, 2.0x] multiplier)

Why 45/55 (not 50/50 or 100/0)

The 45/55 split biases slightly toward workflow fees to reflect the reality that the bulk of Aerchain's value delivery is in execution, not platform access. A client deploying the platform but not running sourcing events gets less value — so they pay less (via the volume multiplier). Competitors default to 100% platform fees (pure subscription) or 100% transaction fees (pure usage). Aerchain's hybrid captures both, which is why total cost to client lands 10-25% below comparable S2P tools at equivalent scope.

5User Multiplier (Platform Access)

Platform Access scales with the client's procurement team size relative to their tier benchmark. Power users count more than light users (they drive more platform load).

weightedUsers    = powerUsers × 1.0 + lightUsers × 0.2
weightedDefault  = tier.powerUsers × 1.0 + tier.lightUsers × 0.2
userRatio        = weightedUsers / weightedDefault
userMultiplier   = clamp(0.85, 1.15, userRatio ^ 0.3)

platformAccess   = (baseline × 0.45) × userMultiplier

6Volume Multiplier (Workflow Fees)

Workflow Fees scale with actual monthly transaction volume relative to the expected volume for the deal's scope preset. Unlike the user multiplier, this has a wider [0.85, 2.00] clamp — heavy-volume clients can pay up to 2x the baseline workflow fee.

expectedVol    = preset.volumeBasis.refVolume × (spend / refSpend) ^ 0.60
volumeRatio    = actualVol / expectedVol
volumeMult     = clamp(0.85, 2.00, volumeRatio ^ 0.30)

workflowFees   = (baseline × 0.55) × volumeMult

Why the asymmetric clamp

Lower bound is tight (−15%) because even a lightly-used deployment still consumes platform resources — Aerchain can't give more than 15% off. Upper bound is wider (+100%) because a client running 10x the volume legitimately costs more to serve. The ^0.3 dampening compresses the curve: 10x volume → 2x multiplier, not 10x. Reflects economies of scale in AI compute and orchestration.

7Volume Discount Curve

On top of the spend-tier baseline, an auto-applied volume discount scales continuously with spend using a smooth power curve:

t              = clamp(0, 1, (spend − $100M) / ($3B − $100M))
volumeDiscount = 70% × t ^ 0.70

// At spend ≤ $100M: 0% discount
// At spend ≥ $3B:   70% (capped)
// Between:          smooth power curve, no cliffs

Why a smooth power curve (not anchor steps)

A step-function discount creates arbitrary cliffs — a $999M client pays dramatically less than a $1.001B client. A sharp CFO will notice and object. The power curve eliminates cliffs entirely: every dollar of additional spend increases the discount smoothly. The ^0.70 exponent is slightly concave, meaning the discount ramps up meaningfully in the mid-range (250M-1B) and then continues scaling toward the cap.

9Implementation with Spend Scaling

Implementation services are calculated from a role-level rate card, multiplied by a tier factor, and finally by a spend scaling factor that kicks in above the Large Enterprise threshold:

baseImpl       = Σ (role.headcount × role.monthly_rate × role.duration)
tierMult       = 0.60 (Mid-Market) / 0.85 (Enterprise) / 1.00 (Large Enterprise)
implSpendMult  = clamp(1.0, 3.0, (spend / $750M) ^ 0.50)

services       = baseImpl × tierMult × implSpendMult

Why square root (diminishing returns)

A $3B rollout is ~2x the effort of a $750M rollout, not 4x. Large Enterprise deployments benefit from economies of scale: reused architecture patterns, same PM team, standardized tooling. The square root (exponent 0.5) captures this realistically — meaningful scaling without linear runaway. Cap at 3.0x prevents services from ballooning at extreme scales ($10B+).

10Gain Share (Future Layer)

Aerchain's pricing model includes a dormant outcome-based layer — Gain Share — for when the platform is ready to commit to measurable savings guarantees. Off by default. When enabled:

gainShareFee = estimatedSavings × defaultPct  (capped at 2× subscription)

The methodology is documented in a separate configurable markdown file and covers three categories: negotiation savings (supplier quote vs. final), cycle time value (reduced procurement lead time × daily value), and maverick spend elimination (redirecting off-contract spend to approved suppliers). All three have defined baselines, measurement methodologies, and audit rights.

Strategic positioning

Gain Share is built into the framework but disabled by default. This lets Aerchain sign deals today at fixed-fee pricing while preserving the option to layer in outcome-based upside as the platform matures and savings measurement becomes more defensible. Infosys partners can propose gain share to clients that specifically request outcome-based structures — without re-architecting the pricing model.

12Business Defensibility Q&A

Why is the pricing BPS so different between $500M and $3B clients?
Because the BPS curve is explicitly designed to reward volume. A $500M client pays ~8 BPS. A $3B client pays ~3 BPS. This is the combination of the declining baseline BPS (10 → 7.5 across tiers) and the volume discount curve (0% → 70% across the same range). Every enterprise SaaS vendor does this — Aerchain just does it with a smooth formula instead of a step-function, so there are no arbitrary cliffs between tiers.
How is the Platform Access fee justified if it doesn't scale directly with users?
Platform Access covers fixed infrastructure costs that don't scale with seat count: AI model inference, integration orchestration, security and compliance, SLA commitments, customer success, standard ERP connectors. These are Aerchain's cost-of-service regardless of whether the client has 10 users or 1,000. The soft user multiplier (±15%) reflects incremental training and provisioning costs without turning this into a per-seat license model.
Why are per-transaction prices back-derived instead of published?
Published per-transaction prices create two problems: (1) they invite apples-to-apples comparison with legacy tools that price per transaction linearly, which misses the AI and orchestration value Aerchain delivers, and (2) the "correct" per-transaction price depends entirely on the deal scope — autonomous sourcing at $40/txn for a narrow-scope deal vs. $4/txn for a full-scope deal is the same total revenue but reads completely differently to a procurement leader. Back-deriving from the workflow budget lets us present per-txn numbers that are deal-appropriate, not arbitrary catalog prices.
Can we offer gain share pricing to a specific client?
Yes. The Gain Share layer is toggleable per deal in the calculator. When enabled, it applies a configurable percentage to measured savings (negotiation, cycle time, maverick spend elimination) with a default cap of 2× subscription fee. The methodology is defined in a separate markdown config that Aerchain sales leadership reviews per deal — it's not a blanket rule. Use this for clients specifically requesting outcome-based structures; default deals stay on fixed-fee pricing.
What if the client wants to validate the numbers independently?
Send them the config file and the formulas in this document. Every number the calculator produces can be reproduced in Excel given the same inputs. There is no AI, no hidden logic, no per-account adjustments. Auditability is a core design principle — deterministic math is the foundation.
Why doesn't the volume discount apply to implementation services?
Implementation is a cost-of-delivery line item, not a usage-based fee. Discounting it would mean Aerchain absorbs the cost difference, which doesn't scale. Services are priced at labor cost (headcount × rate × duration) with a tier multiplier and the new spend scaling factor. If a client pushes hard on implementation pricing, the right response is to reduce scope (toggle sections off) or negotiate milestone payment terms — not discount the hourly math.
How often do the parameters change?
Sales leadership reviews quarterly. Channel complexity weights and spend tier anchors are stable (changed maybe 1-2x per year). Volume discount parameters and tier user defaults are tuned based on closed-deal feedback (maybe 3-4x per year). Implementation rate card changes as salary bands shift. All changes are version-stamped in the config file and tracked in the audit log.
