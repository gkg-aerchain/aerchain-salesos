AI & LLM Model Governance

How Aerchain's AI Works

Aerchain is an AI-native platform. This section addresses the questions Iron Mountain's InfoSec team will have about model training, data usage, tenant isolation in the AI layer, and explainability.
Customer data is never used for model training
Iron Mountain's sourcing data, supplier data, pricing data, and transaction data are never used to train, fine-tune, or improve any shared or public AI models. This is a hard architectural constraint, not a policy toggle. Customer data exists only within the customer's tenant boundary.
CONFIRMED — NO TRAINING ON CUSTOMER DATA
Complete AI tenant isolation
Each customer's AI processing is logically isolated. AI agents operating on Iron Mountain's data cannot access, reference, or be influenced by any other customer's data. Model context windows are scoped per-tenant, per-transaction.
TENANT-SCOPED AI EXECUTION
AI Architecture Detail
QuestionAnswer
What AI/LLM models does Aerchain use?Proprietary agentic AI framework with multi-agent orchestration. LLM capabilities leverage enterprise-grade API providers with data processing agreements in place. No customer data sent to consumer-grade LLM services.
Where does AI inference happen?All AI inference runs on Aerchain's own cloud infrastructure (AWS). No customer data is processed on third-party inference endpoints without explicit DPA and security review.
Is there cross-tenant learning?No. Each tenant's AI operates within its own data boundary. Category intelligence and negotiation models are trained per-tenant. No data or model weights are shared across customers.
How are AI decisions made explainable?Every AI agent decision includes a structured rationale: input data, decision logic, confidence score, and output. Full audit chain from intake to award. Human-readable explanations available for every supplier selection, bid ranking, and negotiation move.
Can Iron Mountain disable AI autonomy?Yes. The platform supports granular autonomy controls. AI can be set to recommend-only (human approves) or fully autonomous per category, per threshold, per region. Configurable at any time.
How is AI bias mitigated?Evaluation criteria are defined by Iron Mountain's procurement team, not by the AI. The AI executes against defined parameters (price, lead time, compliance) with weighted scoring. Scoring logic is transparent and auditable.
What happens if the AI makes an error?All AI decisions are logged with full rollback capability. In Tactical mode, a human reviews every recommendation before it becomes binding. In Autonomous mode, policy guardrails enforce threshold limits and escalation rules.
Data In → AI Processing
ServiceNow SPO metadata + Oracle supplier data enters the AI layer. Data is scoped to the active transaction only. No persistent AI memory across unrelated transactions unless explicitly configured.
Agent Execution
Five specialised agents process sequentially. Each handoff is logged. No agent has access to data outside its designated scope. Context windows are cleared between transactions.
Decision Out → Audit
AI output (supplier recommendation, award decision, negotiation result) is written to the immutable audit log with full rationale before any downstream action (ServiceNow writeback) is triggered.
